# رأيي في التطبيق + ماذا تعمل الآن لبدء بيع الاشتراكات

## رأيي في التطبيق حتى الآن

### نقاط قوة
- **وظائف محاسبية كاملة:** مبيعات، مرتجعات وإلغاء، مخزون، فواتير، مصروفات، تقارير، تصدير Excel للمحاسب، جرد يومي.
- **معمارية بيانات واضحة:** Supabase كقاعدة أساسية + localStorage كـ cache، وطبقة API واحدة (`supabaseApi.js`).
- **أمان:** RLS جاهز، كلمة مرور الأدمن لا تُرفع للسحابة، توثيق أمان و rate limiting و observability.
- **متعدد الحسابات:** دعم أكثر من حساب مع مزامنة سحابية.
- **جاهز للجوال:** PWA + Capacitor (أندرويد)، وواجهة تعمل على الموبايل.
- **اشتراكات في الواجهة:** صفحة اشتراكات وخطة دفع (شهري/سنوي) مع تكامل مبدئي لرابط الدفع (Edge Function).

### ما يمكن تحسينه لاحقاً (ليس شرطاً للانطلاق)
- **ربط الدفع الفعلي:** وجود استدعاء `create-payment` وواجهة الاشتراكات، لكن يحتاج تشغيل Edge Function وربطها ببوابة دفع (PayMob / Accept / فودافون كاش يدوي).
- **فرض الاشتراك:** حالياً التطبيق لا يمنع الاستخدام بدون اشتراك؛ يمكن لاحقاً تقييد بعض الصفحات أو عدد الحسابات حسب الخطة.
- **Observability:** إضافة Sentry أو غيره عند النمو.
- **تجربة أول استخدام:** توجيه واضح (onboarding) للمستخدم الجديد بعد التسجيل أو الدفع.

## ماذا تعمل **الآن** لبدء بيع الاشتراكات

ترتيب عملي مقترح:

### 1) إنشاء جدول الاشتراكات في Supabase (إن لم يكن موجوداً)

من **SQL Editor** في Supabase:

```sql
CREATE TABLE IF NOT EXISTS public.subscriptions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  plan_id TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active',
  started_at TIMESTAMPTZ DEFAULT now(),
  expires_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE public.subscriptions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "users_own_subscriptions" ON public.subscriptions
  FOR ALL
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());
```

(تعديل أسماء الأعمدة إن اختلفت عن ما يستخدمه `subscriptionApi.js`.)

### 2) تفعيل مسار الدفع (أحد الخيارين)

**أ) دفع آلي (مستحسن مع الوقت)**  
- إنشاء **Edge Function** باسم `create-payment` في Supabase تستقبل `plan_id` وترجع `payment_url` من بوابة دفع (مثلاً PayMob أو Accept في مصر، أو Stripe عالمياً).  
- بعد الدفع الناجح: استدعاء نفس الدالة أو webhook لتحديث جدول `subscriptions` (إضافة صف جديد أو تحديث `status` و `expires_at`).  
- في التطبيق الصفحة الحالية (Subscriptions) جاهزة لاستخدام `createPaymentLink(planId)` ثم التوجيه إلى `payment_url`.

**ب) دفع يدوي (للبدء السريع)**  
- الإبقاء على خيار **واتساب** الموجود في الصفحة: المستخدم يختار الخطة ويُفتح واتساب مع رسالة جاهزة.  
- أنت تستلم الدفع (فودافون كاش / انستا باي) ثم تفعّل الاشتراك يدوياً في Supabase (إدراج صف في `subscriptions` بنفس `user_id` و `plan_id` و `expires_at`).  
- لا تحتاج Edge Function في البداية؛ يكفي جدول `subscriptions` + تحديث يدوي.

### 3) صفحة تسجيل دخول + وصول لصفحة الاشتراكات

- التأكد أن **تسجيل الدخول (Supabase Auth)** مطلوب للوصول للتطبيق (أو على الأقل لصفحة الاشتراكات).  
- بعد الدفع (يدوياً أو آلياً) التأكد أن `getCurrentSubscription()` يجد الصف في `subscriptions` حتى تظهر حالة الاشتراك وتاريخ الانتهاء في الواجهة.

### 4) قرار: هل تمنع الاستخدام بدون اشتراك؟

- **الآن:** يمكن ترك التطبيق يعمل بدون اشتراك، والاشتراك يكون "لدعمك" أو "مميزات إضافية لاحقاً".  
- **لاحقاً:** عند استقرار الدفع والخطط، يمكن في الكود التحقق من `isSubscriptionActive(subscription)` ومنع فتح التطبيق أو بعض الصفحات حتى تجديد الاشتراك.

### 5) صفحة هبوط (Landing) بسيطة

- صفحة واحدة توضّح: ماذا يفعل التطبيق، من يستفيد، السعر (شهري/سنوي)، وزر "ابدأ" أو "اشترك" يوجّه للتطبيق أو لصفحة التسجيل ثم الاشتراك.  
- يمكن استضافةها نفس دومين التطبيق (مثلاً `/` أو `/landing`) أو subdomain.

### 6) أشياء قانونية وثقة

- **شروط الاستخدام** و **سياسة الخصوصية** (حتى لو قصيرتان) ووضع رابطهما في التطبيق أو صفحة الاشتراك.  
- **طريقة التواصل والدعم** (واتساب أو إيميل) واضحة للمشتركين.

### 7) نشر التطبيق حيث سيستخدمه الناس

- إن كان ويب: ربط الدومين واستضافة الـ build (Vercel / Netlify / استضافة عربية).  
- إن كان أندرويد: رفع نسخة على Google Play (أو توزيع الـ APK عبر الرابط فقط في البداية).

## ملخص خطوات «الآن» لبدء بيع الاشتراكات

| # | الخطوة |
|---|--------|
| 1 | إنشاء جدول `subscriptions` في Supabase وتفعيل RLS (أعلاه). |
| 2 | اختيار مسار: دفع يدوي (واتساب + تفعيل يدوي في DB) أو دفع آلي (Edge Function + بوابة دفع). |
| 3 | التأكد أن المستخدم يسجّل دخوله وأن صفحة الاشتراكات تعرض حالته بعد إدراج صف في `subscriptions`. |
| 4 | إضافة صفحة هبوط بسيطة + شروط استخدام وخصوصية + قناة دعم. |
| 5 | نشر التطبيق (ويب و/أو أندرويد) والإعلان عن الاشتراكات. |

بعد ذلك يمكنك إضافة فرض الاشتراك على التطبيق، وربط بوابة دفع تلقائي، ثم تحسينات مثل Observability وخطط متعددة (بيسك / بريميوم).
